argument-hint: <실수 또는 새 규칙 설명> (예: "Entity에 비즈니스 로직을 추가했음")
---

AI가 같은 실수를 반복하지 않도록 CLAUDE.md에 규칙을 추가합니다.

**등록할 내용:** $ARGUMENTS

---

## Step 1 — CLAUDE.md 읽기

현재 디렉토리의 `CLAUDE.md`를 읽습니다. 없으면 `.claude/` 또는 상위 디렉토리에서 탐색.

## Step 2 — 규칙 분류

입력 내용을 분석하여 종류를 판단:

| 종류 | 해당 | 등록 섹션 |
|------|------|----------|
| **MUST** | 반드시 해야 하는 패턴 | `반드시 지켜야 할 규칙` |
| **NEVER** | 절대 하면 안 되는 것 | `절대 하면 안 되는 것` |
| **학습된 규칙** | 프로젝트 특화 규칙 | `학습된 규칙 (AI 실수 후 추가)` |

## Step 3 — 규칙 작성

**좋은 예시**
```markdown
- Entity 클래스에 비즈니스 로직(계산, 검증) 추가 금지
  - 이유: 도메인 로직 분산 → 테스트 어려움
  - 대안: Service 레이어에 배치

- 새 API 엔드포인트 추가 시 SpringDoc 어노테이션 필수
  - 이유: API 문서 자동 생성
```

**나쁜 예시**
```markdown
- 코드를 잘 작성할 것         ← 너무 모호
- 항상 주석을 달 것            ← 실행 불가
```

규칙은 **간결 + 실행 가능 + 이유 명시**.

## Step 4 — CLAUDE.md 업데이트

`학습된 규칙 (AI 실수 후 추가)` 섹션에 추가:

```markdown
### YYYY-MM-DD — [실수 유형 한 줄 요약]
- **문제**: 어떤 실수가 발생했는가
- **규칙**: 앞으로 어떻게 해야 하는가
- **이유**: 왜 이 규칙이 필요한가
```

## Step 5 — 줄 수 가드 (≤ 300줄)

규칙을 추가한 직후 CLAUDE.md 줄 수를 확인합니다:

```bash
LINES=$(wc -l < CLAUDE.md)
echo "CLAUDE.md: ${LINES}줄"
```

- **≤ 280줄** — 정상, 다음 단계로
- **281–300줄** — 경고: "캡(300줄)에 근접. 긴 섹션을 `.claude/skills/*.md` 로 이관 권장"
- **> 300줄** — **차단**. 즉시 가장 긴 섹션을 `.claude/skills/{topic}.md` 로 이관하고 CLAUDE.md 에는 ``상세: `.claude/skills/{topic}.md`` 한 줄만 남기도록 사용자에게 제안. 사용자 동의 후 이관 실행.

이유: CLAUDE.md 는 매 세션 로드 — 초과 시 모든 협업자에게 매 세션 토큰 낭비.

## Step 6 — 확인

업데이트 내용을 보여주고 보완 사항 확인.

---

## 언제 쓰나요?

- Claude가 CLAUDE.md 규칙을 어겼을 때
- 코드 리뷰에서 반복 지적사항 발견
- 팀이 새 금지 패턴을 결정
- `/review` 결과 Critical/Major 항목 발생

> **하네스는 실수할 때마다 정교해집니다.** AI가 실수한 직후가 규칙을 추가하기 가장 좋은 타이밍.
