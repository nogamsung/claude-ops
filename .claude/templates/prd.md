## 1. 배경

왜 이 기능이 필요한가? 현재 어떤 문제/불편이 있는가? 한두 문단.

## 2. 목표 (Goals)

- 이 기능이 달성해야 하는 것
- 측정 가능한 지표로 표현 (예: "로그인 성공률 95% 이상", "응답 p95 < 200ms")

## 3. 비목표 (Non-goals)

이번 범위에서 **하지 않는 것**. 후속 이터레이션으로 미루는 항목 포함.

## 4. 대상 사용자

- 주요 사용자 페르소나
- 역할 / 권한

## 5. 유저 스토리

| # | 스토리 | 수락 기준 |
|---|--------|----------|
| US-1 | 사용자로서 X 를 할 수 있어야 한다 | 1) Y 할 때 Z 가 발생 / 2) ... |
| US-2 | ... | ... |

## 6. 핵심 플로우

주요 시나리오를 단계별로 기술. 행복 경로 + 주요 예외 경로.

```
1. 사용자가 X 화면 진입
2. Y 필드 입력 후 제출
3. 서버가 Z 검증 후 A 반환
4. 클라이언트가 B 화면으로 이동
```

### 예외 경로
- 입력 오류 → 인라인 에러
- 서버 오류 → 토스트 + 재시도
- 권한 없음 → 로그인 화면으로 리다이렉트

## 7. 데이터 모델 (요약)

필요한 엔티티와 관계. 세부 스키마는 `/plan db` 로 이어서 설계.

```
User 1 ── N Session
User 1 ── N LoginAttempt
```

## 8. API 계약 (요약)

필요한 엔드포인트. 세부는 `/plan api` 로 이어서 설계.

```
POST /api/v1/auth/login    — 로그인
POST /api/v1/auth/logout   — 로그아웃
GET  /api/v1/me            — 현재 사용자 조회
```

## 9. 비기능 요구사항

| 항목 | 요구 |
|------|------|
| 성능 | 로그인 p95 < 300ms |
| 보안 | 비밀번호 bcrypt, JWT 만료 24h |
| 접근성 | WCAG 2.1 AA, 키보드 네비 |
| i18n | ko / en |
| 로깅 | 로그인 실패 → audit log |

## 10. 의존성 / 리스크

- **의존성**: (예) 기존 User 테이블 스키마, 이메일 발송 서비스
- **리스크**: (예) 레거시 세션 시스템과 병존 기간 발생 → 마이그레이션 전략 필요

## 11. 범위 외 (Out of Scope)

- 소셜 로그인 (후속 PRD)
- 2FA (후속 PRD)
- 비밀번호 재설정 플로우 (별도 PRD)
